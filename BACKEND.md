# Carex — Backend Documentation

## Overview

Carex uses **Supabase** as its entire backend. There is no custom server. Everything — authentication, database, and row-level security — runs through the Supabase hosted platform. The Flutter app communicates directly with Supabase via the `supabase_flutter` SDK.

---

## Authentication

**Provider:** Supabase Auth

All auth logic lives in [`lib/features/auth/repository/auth_repository.dart`](lib/features/auth/repository/auth_repository.dart).

### Sign Up

When a user registers, the app calls `supabase.auth.signUp()` with the user's email and password. Extra fields (name, role, phone, date of birth, gender, medical conditions) are passed in the `data` map as user metadata.

A Supabase **database trigger** (defined server-side) reads this metadata and automatically inserts a row into the `profiles` table when the auth user is created. The `profiles.id` is the same UUID as `auth.users.id`.

After signup, Supabase sends a confirmation email. The app checks if a session was returned:
- **No session** → email confirmation is required. The user sees a message to verify their email.
- **Session present** → the user is logged in immediately.

### Sign In

Email/password login via `supabase.auth.signInWithPassword()`.

OAuth sign-in (Google, Facebook, Apple) is supported via `supabase.auth.signInWithOAuth()`. These open a browser/webview for the OAuth flow and return to the app on completion.

### Password Reset

1. The app calls `resetPasswordForEmail(email)` with a deep link redirect: `carex://reset-password`.
2. Supabase emails the user a link.
3. On click, the deep link opens the app and routes to the reset password screen.
4. The app calls `updatePassword(newPassword)` via `supabase.auth.updateUser()`.

### Session Management

The `supabase_flutter` SDK handles session persistence automatically. The `SplashBloc` checks `supabase.auth.currentSession` on app start to decide whether to send the user to the home screen or the login screen.

## Database

**Engine:** PostgreSQL (via Supabase)

All tables live in the `public` schema. Every table uses `uuid` primary keys generated by `gen_random_uuid()`.

---

### Table: `profiles`

Mirrors `auth.users`. Created automatically by a database trigger on signup.

| Column | Type | Notes |
|---|---|---|
| `id` | uuid | FK → `auth.users.id` (PK) |
| `full_name` | text | |
| `role` | text | `'patient'` or `'doctor'` |
| `phone` | text | |
| `date_of_birth` | text | |
| `gender` | text | |
| `medical_conditions` | text | |
| `fcm_token` | text | Firebase Cloud Messaging token for push notifications |
| `notif_medications` | boolean | Default `true` |
| `notif_missed_dose` | boolean | Default `true` |
| `notif_weekly_reports` | boolean | Default `false` |
| `notif_sos` | boolean | Default `true` |
| `created_at` | timestamptz | |

The `role` field controls what the user can see and do in the app. A `patient` manages their own data. A `doctor` can view the data of linked patients.

### Table: `patient_doctor_links`

A join table that links patients to their doctors. Both `patient_id` and `doctor_id` are FKs to `profiles`.

| Column | Type | Notes |
|---|---|---|
| `id` | uuid | PK |
| `patient_id` | uuid | FK → `profiles.id` |
| `doctor_id` | uuid | FK → `profiles.id` |
| `created_at` | timestamptz | |

This relationship allows a doctor to query medications, readings, adherence logs, and SOS alerts belonging to their linked patients.

| Column | Type | Notes |

| Column | Type | Notes |
|---|---|---|

### Table: `adherence_logs`

Records whether a patient took, missed, or skipped a scheduled dose. One row per dose event.

| Column | Type | Notes |
|---|---|---|
| `id` | uuid | PK |
| `schedule_entry_id` | uuid | FK → `schedule_entries.id` |
| `patient_id` | uuid | FK → `profiles.id` |
| `scheduled_at` | timestamptz | When the dose was due |
| `taken_at` | timestamptz | When the patient actually took it (nullable) |
| `status` | text | `'taken'`, `'missed'`, or `'skipped'` |
| `created_at` | timestamptz | |

**Repository:** [`lib/features/medications/repository/adherence_repository.dart`](lib/features/medications/repository/adherence_repository.dart)

#### Offline Support

`AdherenceRepository` uses **Hive** (a local key-value store) to handle offline logging. If the Supabase insert fails (no internet), the payload is saved to a local Hive box named `pending_adherence_logs`. When connectivity returns, `syncPending()` flushes all queued logs to Supabase in a single batch insert and clears the box.

Operations: `logDose`, `syncPending`, `getLogsForEntry`, `getRecentLogs`, `getLogsForPatient`.

| Column | Type | Notes |
|---|---|---|

| `type` | text | `'BP'`, `'Sugar'`, or `'Pulse'` |
| `value` | text | The reading value (e.g. `"120/80"`) |
| `unit` | text | Auto-set: `mmHg`, `mg/dL`, or `bpm` |

**Repository:** [`lib/features/readings/repository/reading_repository.dart`](lib/features/readings/repository/reading_repository.dart)

The `unit` is derived from the `ReadingType` enum on the Flutter side and stored alongside the value. Doctors can query a specific patient's readings via `getReadingsForPatient(patientId)`.

Operations: `getReadings`, `addReading`, `deleteReading`, `getReadingsForPatient`.

| Column | Type | Notes |
|---|---|---|

## Local Notifications

**Service:** [`lib/core/services/notification_service.dart`](lib/core/services/notification_service.dart)

**Library:** `flutter_local_notifications`

The `NotificationService` is a singleton that schedules daily local notifications for medication reminders. It does **not** call Supabase — it works entirely on-device.

### How it works

1. `init()` is called once at app startup. It initializes the notification plugin and requests permissions (Android and iOS).
2. `resyncSchedule()` is called whenever the medication schedule changes. It cancels all existing notifications and re-schedules one daily repeating notification per `ScheduleEntry`.
3. Each notification fires at the time defined in `schedule_entry.time` (e.g. `"08:00"`). If that time has already passed for today, the first fire is scheduled for tomorrow.
4. The notification payload contains the `scheduleEntryId`. When the user taps the notification, the app receives this ID via the `onNotificationTap` callback and can navigate to the relevant dose screen.
5. Notification IDs are derived from the schedule entry UUID using `hashCode % 2147483647` to fit within Android's int limit.

The `fcm_token` on the `profiles` table is stored for future server-triggered push notifications (e.g. SOS alerts sent to the doctor's device), but FCM push delivery is handled server-side outside the Flutter app.

## Data Flow Summary

```
Patient action (e.g. "Mark dose as taken")
        |
        v
  Flutter UI (BLoC event)
        |
        v
  Repository (e.g. AdherenceRepository.logDose)
        |
     Online? ──── No ──→ Hive local queue
        |                     |
       Yes               (syncPending on next launch)
        |
        v
  Supabase PostgREST API
        |
        v
  PostgreSQL table (e.g. adherence_logs)
```

```
App startup
        |
        v
  SplashBloc checks supabase.auth.currentSession
        |
   Session exists? ── No ──→ Login / Onboarding
        |
       Yes
        |
        v
  Route to Patient Home or Doctor Dashboard
        |
        v
  NotificationService.resyncSchedule() called
  (re-schedules all medication reminders)
```

## Security

Row-Level Security (RLS) should be enabled on all tables in Supabase so that:
- Patients can only read/write their own rows (where `patient_id = auth.uid()`).
- Doctors can read rows belonging to patients linked to them via `patient_doctor_links`.
- No user can read another user's profile unless linked.

The Flutter app does not implement any additional access control — it relies entirely on Supabase RLS policies enforced at the database level.
